deploy.sh Internal Architecture

File: enterprise/deploy.sh (~620 lines). Key design decisions:

  • Docker builds on EC2, not locally: Source tarball → S3 → EC2 pulls → builds ARM image → ECR push. You don\'t need Docker installed. The build uses the same Graviton ARM64 architecture as production.
  • AgentCore Runtime env vars (Step 4): STACK_NAME, AWS_REGION, S3_BUCKET, DYNAMODB_TABLE, DYNAMODB_REGION, BEDROCK_MODEL_ID passed via --environment-variables flag. This is the ONLY way to configure the container — no .env file inside the image.
  • ec2-setup.sh (Step 8): The critical final step. Builds React frontend (npm install + vite build), installs 3 systemd services, and injects AWS_ENDPOINT_URL_BEDROCK_RUNTIME=http://localhost:8091 into Gateway service — connecting Gateway to H2 Proxy for multi-tenant routing.

⚠️ Self-Audit: Deployment Gotchas

  • First deploy on new account may fail (ECS service linked role): New AWS accounts need ~30s for ECS role propagation. First CFN create may fail with "Unable to assume service linked role." Fix: wait, re-run. Second attempt always succeeds.
  • S3 bucket name conflict (multi-region): Default name is openclaw-tenants-{ACCOUNT_ID}. Same account, two regions → conflict. Must set WORKSPACE_BUCKET_NAME.
  • seed scripts need import os: After batch-replacing hardcoded defaults with os.environ.get(), all seed scripts must have import os. Missing this causes silent seed failure — DynamoDB table created but empty (no employees, no roles).
  • JWT_SECRET regenerates on each deploy.sh run: Not stored in .env — generated fresh with openssl rand -hex 32. Re-deploying with --skip-seed still writes new JWT_SECRET to SSM, invalidating existing tokens. Employees must re-login.
  • Docker image must be rebuilt after agent-container code changes: workspace_assembler.py, server.py, permissions.py are baked into the image. Code changes on EC2 admin-console don\'t affect the container — must rebuild with deploy.sh --skip-seed.
